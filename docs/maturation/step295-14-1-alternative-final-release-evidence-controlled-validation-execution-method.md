controlled record-based validation execution method.

Policy v0.1 and Controlled Validation Plan v0.1. It does not execute validation

scope separately identified by committed predecessor evidence.

## 8. Fixed Execution Sequence

Future execution must follow this record-based sequence:

1. Confirm execution authorization identity and method identity.
2. Confirm policy, plan, review, candidate, package, and toolchain continuity.
3. Confirm each required permitted evidence class exists in approved safe form.
4. Apply fixed questions A through K in order.
5. Stop immediately if prohibited evidence, unavailable evidence, identity
   uncertainty, contradiction, or invalidation is encountered.
6. Assign only the pre-defined category-level question statuses.
7. Derive one aggregate validation status through the fixed precedence rule.
8. Record only approved safe category-level results and unresolved limitations.
9. Do not determine policy adoption, alternative-evidence sufficiency, zero
   findings, or public release.

This sequence does not include raw command use, browser operation, external API
execution, Plugin Check execution, Plugin Check output inspection, source-code
inspection, or candidate/package modification.

## 9. Fixed Question-Status and Aggregate-Status Model

Question status values for each question A through K:

- Satisfied
- Failed
- Evidence unavailable
- Invalidated
- Blocked

Aggregate validation status values:

- Validation passed
- Validation failed
- Evidence unavailable
- Invalidated
- Blocked

Precedence rule:

1. Invalidated: any required identity, continuity, method, policy, plan,
   toolchain, disclosure, or role boundary is invalidated.
2. Blocked: no invalidation applies, but an execution prerequisite,
   authorization, role boundary, or safe classification boundary is missing.
3. Evidence unavailable: no invalidation or blocker applies, but a required
   condition cannot be safely established without prohibited, raw, sensitive,
   private, ambiguous, or unsupported material.
4. Validation failed: no higher-precedence outcome applies, but a required
   approved condition is contradicted, incomplete, or outside approved scope.
5. Validation passed: every fixed question A through K is Satisfied using only
   approved safe category-level evidence, with no prohibited evidence or
   inference.

Validation passed does not establish:

- Alternative release-evidence sufficiency.
- Policy adoption.
- Zero findings.
- Resolution of strict Plugin Check aggregate evidence.
- WordPress.org release approval.

For every status:

WordPress.org public release readiness:

Hold

Policy adoption:

Not automatically authorized

Final release decision:

Not automatically authorized

## 10. Stop Conditions and Fail-Closed Boundary

Future method execution must stop without inference if any of the following
occur:

- A prohibited evidence type becomes required.
- A required identity is ambiguous or mismatched.
- A candidate/package/toolchain/policy/plan/method continuity boundary is lost.
- The independent review or execution authorization is absent or invalid.
- Strict Plugin Check and alternative evidence become conflated.
- A required input cannot be safely classified.
- A category-level result would require raw, sensitive, private, or unsupported material.
- A role-separation boundary cannot be maintained.
- A newly eligible public supported Plugin Check contract becomes available.

If a stop condition occurs, future execution must use the applicable Evidence
unavailable, Invalidated, or Blocked status and keep release readiness at Hold.

# Step 81: Credential / OAuth Strategy Decision Checkpoint

## Purpose

This document records the Step 81 credential and OAuth strategy decision
checkpoint for Analytics Report AI.

Step 80 identified credential storage, manual Google Access Token entry,
Google OAuth/token lifecycle, OpenAI API Key storage, support/debug redaction,
and uninstall cleanup as release blocker candidates. This checkpoint narrows
the review to those credential-related decisions and prepares recommended
strategy candidates for later human decision and implementation planning.

This is not an implementation step, not a `readme.txt` update, not an OAuth
implementation, not a credential storage change, and not a release-readiness
decision.

WordPress.org release remains `Hold`.

## Google Authentication Strategy Options

### Option A: Keep Manual Google Access Token Entry For Public Release

Pros:

- Lowest implementation cost.
- Preserves the current MVP flow.
- Useful for trusted local developer verification.

Cons:

- Users must obtain a token outside the plugin.
- Tokens expire and reconnect UX is not built.
- Scope and property access are not validated as a connection experience.
- Revocation/disconnect behavior is manual and unclear.
- Support burden is high because token setup errors are likely.

Security/privacy concerns:

- A live Google credential category is stored in WordPress database settings.
- Database administrators, backups, server administrators, and code able to
  read options may access the saved credential.
- Support screenshots or option dumps could leak secrets if public guidance is
  weak.

UX/support concerns:

- Manual token generation is not a normal public plugin setup flow.
- Expired token, insufficient permission, and property access failures are hard
  for non-developer users to resolve.

Token expiry concerns:

- Short-lived access tokens make the setup fragile.
- Without refresh/reconnect behavior, report generation may fail unexpectedly.

WordPress.org release concern:

- This option should be treated as a release blocker candidate.

Recommendation:

- Do not treat manual Google Access Token entry as public-release ready.

### Option B: Keep Manual Token Only For Developer Verification, Block WordPress.org Release Until OAuth Strategy Exists

Pros:

- Matches the current MVP disclosure that manual token entry is temporary.
- Avoids presenting a developer-oriented flow as a public-ready feature.
- Allows the current local verification workflow to remain useful while release
  planning continues.
- Keeps WordPress.org release on hold until the authentication story is safer
  and clearer.

Cons:

- Delays public release.
- Requires additional planning before release readiness can resume.
- Does not itself solve OAuth, token lifecycle, or storage issues.

Implementation impact:

- No immediate production code change in this checkpoint.
- Later implementation may either remove, hide, restrict, or relabel manual
  token entry depending on the chosen release strategy.

Documentation impact:

- Future readme/admin/privacy wording should state that manual entry is for
  developer verification only unless a different public strategy is accepted.

Release impact:

- WordPress.org release remains blocked until OAuth/token lifecycle strategy is
  explicitly designed and accepted or implemented.

Recommendation:

- Recommended as the current decision candidate.

### Option C: Implement Full OAuth Before Public Release

Pros:

- Provides a normal user-facing Google connection flow.
- Creates a path for explicit scopes, expiry handling, reconnect, and
  disconnect/revoke UI.
- Reduces user confusion around manual token generation.
- Improves release disclosure and support posture.

Cons:

- Larger implementation and QA scope.
- Requires careful security design for callback state, redirect behavior,
  capability checks, token exchange, storage, and revocation.
- May require Google app setup, verification, or a site-owner configuration
  model.

Required implementation areas:

- OAuth authorization initiation.
- OAuth callback handling with state protection.
- Administrator capability checks.
- Redirect URI validation.
- Token exchange.
- Access token expiry tracking.
- Refresh behavior if refresh credentials are used.
- Scope validation.
- Property access validation or clear failure handling.
- Reconnect UI.
- Disconnect/revoke UI.
- Safe error messages.
- Support/debug redaction guidance.
- Uninstall cleanup policy.

Token lifecycle requirements:

- Expiry tracking.
- Refresh behavior or explicit reconnect flow.
- Invalid/revoked credential detection.
- Insufficient permission handling.
- Property access failure handling.

Support impact:

- Support can focus on connection status and safe error categories instead of
  manual token creation.
- Support documentation still must prohibit sharing secrets, option dumps,
  request/response bodies, full payloads, and generated report bodies.

Release impact:

- This is the strongest public-release path, but it requires a dedicated
  design, implementation, and QA phase.

Recommendation:

- Preferred long-term public-release direction.

### Option D: Remove Google Token Entry From Public Release Scope And Keep Plugin Unreleased / Private Until Authentication Is Redesigned

Pros:

- Avoids shipping a known developer-oriented authentication model.
- Keeps credential and support risks contained to private/developer contexts.
- Provides time to choose between site-local OAuth, hosted broker, or another
  authentication model.

Cons:

- No WordPress.org release path until authentication is redesigned.
- Product value remains limited to private verification.
- May require communication that the MVP is intentionally not public-ready.

Product impact:

- Keeps the current plugin useful for controlled developer verification, but
  not for broad public distribution.

Release impact:

- WordPress.org release remains on hold.

Recommendation:

- Acceptable if the team is not ready to implement OAuth or another public
  authentication model soon.

## OpenAI API Key Strategy Options

### Option A: Continue Storing OpenAI API Key In Plugin Settings

Pros:

- Preserves current MVP behavior.
- Simple for administrators to configure.
- Existing non-redisplay and delete controls can be preserved.

Cons:

- The key category is stored in the WordPress database.
- Database administrators, backups, server administrators, and option-reading
  code may access the stored key.
- Public support must be very clear about never sharing secrets or option
  dumps.

Security/privacy concerns:

- Persistent settings-based key storage is easier to use but has database and
  backup exposure.
- A compromised administrator account or vulnerable code with option access can
  expose the key category.

UX/support concerns:

- Familiar Settings UI is easy for site owners.
- Support must explain restricted keys, permissions, non-redisplay, and safe
  deletion.

Release concern:

- May be acceptable only with explicit disclosure and a documented support
  redaction policy.

Recommendation:

- Needs explicit human acceptance before public release.

### Option B: Support Constant-based Configuration For OpenAI API Key

Pros:

- Allows site owners to keep the key outside the WordPress options table.
- Works well for managed hosting, deployment pipelines, and private config.
- Reduces database backup exposure.

Cons:

- Less friendly for non-technical administrators.
- Requires clear documentation and UI status behavior.
- Does not eliminate all server-level exposure.

Implementation impact:

- Add constant lookup path and safe precedence behavior.
- Ensure the UI indicates configuration status without exposing the value.
- Ensure settings save cannot overwrite or reveal constant-based secrets.

Documentation impact:

- Document where and how a site owner configures the constant.
- Explain that the value is still sensitive server configuration.

Release impact:

- Strong candidate before public release because it improves deployment
  options and reduces option-table exposure.

Recommendation:

- Consider implementing before public release.

### Option C: Support Both Settings-based And Constant-based Configuration

Pros:

- Keeps easy setup for smaller sites.
- Supports safer managed deployments.
- Allows a gradual migration path.

Cons:

- Requires explicit precedence rules.
- UI and support documentation become more complex.
- Needs careful handling so a settings value is not shown or used
  unexpectedly when a constant is configured.

Precedence rules:

- Constant-based configuration should likely take precedence if present.
- Settings UI should show status only and explain that the key is managed by
  site configuration.
- Delete controls should affect only settings-based storage, not constants.

UI disclosure requirements:

- Display configured-via-site-configuration status without showing the key.
- Explain which storage path is active.
- Preserve no-redisplay behavior for all key sources.

Support impact:

- Support can recommend constant-based setup for advanced users.
- Support docs must explain precedence without asking users to paste secrets.

Release impact:

- Balanced public-release candidate if documented and tested.

Recommendation:

- Recommended candidate for planning, pending implementation scope.

### Option D: Require Site Owner-managed Key Outside Plugin UI

Pros:

- Avoids storing the OpenAI API Key in plugin settings.
- Encourages deployment-managed secret handling.
- Reduces accidental option dump exposure.

Cons:

- Higher setup barrier.
- May be less suitable for typical WordPress.org users.
- Requires clear failure messages when no configured key exists.

UX impact:

- More technical setup.
- Settings screen would need to guide users without collecting the key.

Support impact:

- Support shifts to configuration guidance.
- Support must still prohibit sharing secrets.

Release impact:

- Strong security posture but may reduce usability for public directory users.

Recommendation:

- Consider as a private/managed-deployment mode, or as a future stricter
  release option if settings-based storage is rejected.

## Recommended Strategy Candidate

This checkpoint proposes the following candidate for later human decision and
implementation planning. It is not an implementation decision by itself.

```text
Google authentication: Manual Google Access Token entry should remain developer-verification only. WordPress.org release should remain blocked until OAuth / token lifecycle strategy is explicitly designed and accepted or implemented.
OpenAI API Key: Settings-based storage may be acceptable only with explicit disclosure and support redaction policy, but constant-based configuration should be considered before public release.
Credential storage: Current MVP storage should not be treated as release-ready without an explicit acceptance/redesign decision.
Uninstall cleanup: Credential-bearing settings cleanup policy must be decided before release readiness.
Support/debug: Public support must forbid sharing secrets, option dumps, request/response bodies, full payloads, and generated report bodies.
Release position: Hold.
```

Rationale:

- The current Google manual token model is useful for developer verification
  but too fragile for public distribution.
- OAuth/token lifecycle affects readme wording, admin copy, error-path QA,
  support policy, and uninstall cleanup.
- OpenAI key handling has a narrower lifecycle problem than Google auth, but
  still needs a public storage strategy.
- Support/debug safety must be designed before public users can be asked for
  troubleshooting evidence.
